Consistency Issue

View Toggle Unification

UX rationale: Three pages, three different toggle implementations. Users build muscle memory from consistent controls. When toggles look and behave differently on each page, it erodes trust and slows navigation.

Before — Three different implementations

Talent Page

Custom <div> buttons, border container, gray active state

Raw <button> elements with inline className toggling. Uses LayoutGrid + LayoutList icons. Active = bg-surface-subtle.

Locations Page

Button component, variant toggling, gap-1

Uses Button component. variant="default" (black bg) for active, variant="outline" for inactive. LayoutList + Table2 icons.

Shots Page

Button component + keyboard hint overlays

Uses Button component with keyboard hints (1/2). LayoutGrid + Table2 icons. ml-auto positioning.

After — Single ViewModeToggle component

Implementation: Extract ViewModeToggle to shared/components/ViewModeToggle.tsx. Props: modes (array of {key, icon, label}), activeMode, onModeChange. Uses Button component consistently.

Talent — Grid / Table

Locations — List / Table

Shots — Card / Table

Consistent pattern: Button variant="default" (filled) for active, variant="outline" for inactive. Icon convention: LayoutGrid = grid/card, Table2 = table, LayoutList = list. No keyboard hints (handled by global shortcuts).

Consistency Issue

Search Bar Standardization

UX rationale: Four library pages use search inputs, but they are styled differently. Talent has no search icon, Crew and Locations have icons but different container widths. Users expect visual consistency for the same interaction pattern.

Before — Inconsistent search inputs

Talent Page

No search icon. Container = max-w-md flex-1. Plain Input component with no icon affordance. Less discoverable.

Crew Page

Has search icon. Container = max-w-sm (no flex-1). No clear button.

Locations Page

Has search icon. Container = max-w-sm flex-1. No clear button.

Shots Page

Has search icon + clear button. Fixed width w-[260px]. This is closest to ideal.

After — Unified SearchInput component

Implementation: No new component needed. Standardize the Input pattern across all pages: always use the search icon + pl-9 pattern. Add clear button when query is non-empty. Container: relative max-w-sm flex-1.

All pages use identical search pattern:

Pattern: Search icon (left), Input with pl-9, X clear button (right, visible when query non-empty). Container: relative max-w-sm flex-1. Same on Talent, Crew, Locations, Shots pages.

After — Single unified notes pattern

Implementation: Create shared/components/UnifiedNotes.tsx with a single rendering pipeline. Input: value: string | null. Decision tree:

Rendering Decision Tree

1.

Is value null/empty?

→ Show placeholder: "No notes yet" or "Click to add notes..."

2.

Does value contain HTML tags? /<\w+[^>]*>/

→ Read mode: render via SanitizedHtml component (strips dangerous tags, preserves formatting)

3.

Plain text value?

→ Read mode: render with whitespace-pre-wrap + URL linkification

4.

Edit mode (all content types)?

→ Always use plain Textarea. New notes are always stored as plain text. Legacy HTML is read-only.

Implementation notes:

  • Talent: replace InlineTextarea for notes with UnifiedNotes (or inline the HTML detection logic)
  • Locations: replace raw Textarea with the same read/edit pattern used by NotesSection
  • The NotesSection pattern from shots/crew is the gold standard — generalize it
  • Legacy HTML notes are NEVER edited — always render read-only via SanitizedHtml
  • New notes are always saved as plain text (no HTML editor needed)

~2 hours
