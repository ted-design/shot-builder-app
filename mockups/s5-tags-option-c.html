Design Notes

  • Per-tag granularity — each tag gets its own color. "High Priority" (red) looks different from "Low Priority" (green).
  • Uses existing data — the Firestore color field on each tag is already populated (preserved during Sprint S4 neutralization).
  • Very subtle tints — bg at 8% opacity, border at 22% opacity. Keeps the editorial restraint of Direction A.
  • Text stays high-contrast — dark text on light mode, light text on dark mode. Uses Tailwind-700 (light) / 300 (dark) levels.
  • Sort order: priority → gender → media → other. Within each category, alphabetical.
  • Size: text-xxs (11px), rounded-md, px-2 py-0.5. Same compact footprint.
  • Implementation: Reuse tagColors.ts palette. Convert TAG_COLOR_CLASS_MAP to use rgba-based tints instead of Tailwind utility classes. Or use inline styles with CSS custom properties: style="--tag-rgb: r,g,b" and bg-[rgba(var(--tag-rgb),0.08)].
  • Pros: Most informational — each tag is visually unique. Leverages existing Firestore data. Users with custom tags can pick meaningful colors.
  • Cons: More implementation work (need rgb values per color key). Could look rainbow-ish with many tags on one card (but 8% opacity keeps it subtle).

UX Researcher Recommendation

Option C (Per-Tag Custom Color) provides the best balance of information density and editorial restraint. The 8% opacity tints are subtle enough not to clash with the near-black editorial direction, while providing meaningful visual differentiation. It also leverages existing Firestore data that was specifically preserved during Sprint S4 for this future use. Combined with category-based sorting, it gives users both color and spatial grouping cues.

Option B (Left Border) is the safest fallback if Option C feels too colorful after visual testing. Option A works but has less granularity (all priority tags look the same regardless of urgency level).
